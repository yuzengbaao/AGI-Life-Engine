# AGI 演化协议 (The Evolution Protocol)

> "试错不是混乱，而是进化的必经之路。但为了高效进化，我们需要一套‘基因编辑规范’。"

**日期**: 2025-12-12
**版本**: 1.0
**背景**: 鉴于 AGI 开发过程中频繁出现的“试错”与“不确定性”，本协议旨在将这种探索过程标准化，从“随机突变”转变为“定向进化”。

## 1. 核心认知：开发即进化

我们承认：**AGI 不是被“写”出来的，而是被“养”出来的。**
每一次代码修改，实际上是在调整它的基因（System Code）或环境（Context），然后观察它的表型（Behavior）。因此，我们不再追求“一次写对”，而是追求“快速、安全的迭代闭环”。

## 2. 标准化演化流程 (The Cycle)

为了避免混乱，所有的功能修改必须遵循 **H-I-V-E** 循环：

### 2.1 H - Hypothesis (假设)
*   **定义**: 在修改代码前，明确预期的行为改变。
*   **格式**: "如果我修改 [模块X]，AGI 应该能够 [新行为Y]。"
*   **例子**: "如果我强制 `SystemTools` 使用 JSON 格式，AGI 的文件操作成功率应提高到 95% 以上。"

### 2.2 I - Implementation (实现)
*   **原则**: **最小干预 (Minimal Intervention)**。
*   **操作**:
    1.  优先修改 `System Prompt` 或 `Work Templates` (软调整)。
    2.  其次修改 Python 逻辑代码 (硬调整)。
    3.  严禁在没有备份的情况下直接修改核心引擎 (`AGI_Life_Engine.py`)。

### 2.3 V - Verification (验证)
*   **原则**: **可观测性 (Observability)**。
*   **操作**: 修改后必须立即运行一个“探针任务”来验证。
    *   *Bad*: "改好了，你试试。"
    *   *Good*: "已修改。正在运行 `user_command.txt` 中的测试指令：`Task: Verify File Write`。请查看 `watch_consciousness.py` 的输出。"

### 2.4 E - Evolution / Rollback (进化/回滚)
*   **成功**: 更新 `PROJECT_STATUS.md`，将特性固化。
*   **失败**: 立即回滚代码，并记录失败原因到 `docs/errors/` (积累失败知识库)。

## 3. 交互规范：人机协作

为了减少沟通损耗，确立以下协作模式：

*   **用户 (User)**: 扮演“自然选择”的角色 (Natural Selection)。
    *   提供生存压力 (提出需求)。
    *   裁决生存权 (确认是否满意)。
*   **开发者 (You)**: 扮演“基因工程师”的角色 (Genetic Engineer)。
    *   负责剪辑代码。
    *   负责解释变异的后果。
*   **AGI (System)**: 扮演“实验体”的角色 (Subject)。
    *   在沙箱中运行，展示能力。

## 4. 当前痛点与改进计划

| 痛点 | 改进方案 | 状态 |
| :--- | :--- | :--- |

## 5. 结语

我们不再畏惧试错。我们将试错**制度化**。
每一次报错，都是 AGI 撞击现实边界的回声；每一次修复，都是它向强智能迈进的一小步。
